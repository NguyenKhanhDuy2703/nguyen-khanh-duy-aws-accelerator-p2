Bên trong kiến trúc Kubernetes: từ Overview đến từng thành phần nội bộ

Tài liệu này phân rã Kubernetes thành các lớp: Control Plane, Worker Node, Pod, Network và Storage — mỗi phần đi kèm sơ đồ và giải thích luồng dữ liệu/điều khiển thực tế giữa các thành phần.

Control Plane kubelet / kube-proxy etcd CNI / CSI / CRI Scheduling Reconciliation Loop

01Overview kiến trúc tổng thể

Kubernetes là hệ thống phân tán theo mô hình declarative (khai báo trạng thái mong muốn) và controller-based reconciliation (vòng lặp đưa trạng thái thực tế khớp với trạng thái mong muốn). Toàn bộ cluster chia thành 2 mặt phẳng: Control Plane (bộ não, quản lý trạng thái) và Data Plane / Worker Nodes (nơi container thực sự chạy).

Sơ đồ 1 — Tổng quan kiến trúc Cluster
Mọi tương tác trong cluster — từ kubectl apply đến việc kubelet báo cáo trạng thái Pod — đều đi qua API Server. Nó là "cửa ngõ" duy nhất tới etcd, nơi lưu trữ toàn bộ trạng thái cluster.

Control Plane Não bộ của Cluster

Quyết định cái gì nên xảy ra: lưu trạng thái mong muốn, lên lịch Pod, theo dõi và sửa sai lệch. Thường chạy trên các Node riêng (master nodes), có thể HA với nhiều bản sao.

Worker Node Nơi việc thực thi

Thực thi những gì Control Plane yêu cầu: chạy container, quản lý network rule, mount volume, báo cáo metric/health liên tục.

02Phân cấp đối tượng trong Cluster

Trước khi đi sâu vào từng tiến trình, cần hiểu cách Kubernetes tổ chức các API object — đây là "ngôn ngữ" mà mọi thành phần dùng để giao tiếp.

Sơ đồ 2 — Cây phân cấp Workload Object
Deployment → ReplicaSet → Pod → Container là chuỗi "ownership" — mỗi cấp tạo và quản lý cấp dưới qua ownerReferences. Service không nằm trong chuỗi này mà "trỏ" tới Pod bằng label selector, tách rời địa chỉ mạng khỏi vòng đời Pod (Pod chết đi sống lại, Service vẫn giữ địa chỉ ổn định).

03Control Plane — Phân tích chi tiết

Control Plane gồm 4 thành phần lõi (+ 1 tuỳ chọn). Chúng đều là stateless (trừ etcd) và chỉ giao tiếp với etcd thông qua API Server — không bao giờ truy cập etcd trực tiếp.

Sơ đồ 3 — Luồng dữ liệu nội bộ Control Plane
kube-apiserver — cổng vào/ra duy nhất Watch (long-poll, push khi có thay đổi) Write (tạo/cập nhật object)

3.1 kube-apiserver — "Cổng giao tiếp duy nhất"

API Server là một HTTP REST server stateless, expose toàn bộ API Kubernetes (Pods, Services, ConfigMaps...). Mọi request đi qua 3 giai đoạn tuần tự:

1
Authentication (AuthN)

Xác định "ai đang gọi" — qua client certificate, bearer token (ServiceAccount), OIDC token...

2
Authorization (AuthZ)

Xác định "người này có quyền làm hành động này không" — chủ yếu qua RBAC (Role/ClusterRole + Binding).

3
Admission Control

Chuỗi plugin chạy trước khi object được lưu: Mutating (tự thêm/sửa field, vd inject sidecar) rồi Validating (chấp nhận hoặc từ chối, vd OPA Gatekeeper, PodSecurity).

Sau khi qua cả 3 lớp, object được serialize và ghi vào etcd qua gRPC. API Server cũng duy trì watch cache trong RAM để phục vụ các client watch (scheduler, controller, kubelet) mà không tạo áp lực trực tiếp lên etcd cho mỗi lần đọc.

3.2 etcd — Nguồn sự thật duy nhất

etcd là distributed key-value store dùng thuật toán đồng thuận Raft. Toàn bộ trạng thái cluster — từ Pod, Node, Secret đến ConfigMap — được lưu dưới dạng key dạng:

/registry/{resource}/{namespace}/{name} — ví dụ /registry/pods/default/my-app-7f9c

  • Số node lẻ (3, 5, 7) để đảm bảo quorum khi bầu leader, tránh split-brain.
  • Mọi write phải được majority node xác nhận (commit) mới trả thành công.
  • Đây là thành phần duy nhất stateful trong Control Plane — backup etcd = backup toàn bộ cluster.

3.3 kube-scheduler — Bộ chọn Node

Scheduler chỉ làm một việc: với mỗi Pod mới có spec.nodeName == "", nó chọn Node phù hợp nhất rồi ghi nodeName vào Pod (gọi là binding). Quá trình chọn gồm 2 pha:

Filtering (Predicates)

Loại các Node KHÔNG đủ điều kiện: thiếu CPU/RAM, không khớp nodeSelector/Affinity, Taint không được Toleration chấp nhận, port conflict...

Scoring (Priorities)

Chấm điểm các Node còn lại: ưu tiên Node ít tải hơn (spread), gần dữ liệu hơn (affinity), tổng điểm cao nhất thắng.

3.4 kube-controller-manager — Tập hợp Controller

Đây không phải 1 tiến trình logic mà là vỏ bọc chạy chung nhiều controller trong cùng 1 binary, mỗi controller chạy một reconciliation loop độc lập:

ControllerTrách nhiệm
Node ControllerTheo dõi heartbeat Node, đánh dấu NotReady / evict Pod khi Node chết
ReplicaSet ControllerĐảm bảo số Pod thực tế = spec.replicas
Deployment ControllerQuản lý rolling update giữa các ReplicaSet
Endpoint/EndpointSlice ControllerCập nhật danh sách IP Pod backing cho mỗi Service
Namespace ControllerDọn dẹp resource khi Namespace bị xoá
ServiceAccount/Token ControllerTạo ServiceAccount mặc định + token cho Namespace mới

04Worker Node — Phân tích chi tiết

Mỗi Worker Node chạy 3 tiến trình lõi giao tiếp với nhau qua các giao diện chuẩn hoá (CRI, CNI, CSI) — đây là lý do Kubernetes có thể "cắm" nhiều runtime/network plugin/storage khác nhau mà không sửa code lõi.

Sơ đồ 4 — Nội bộ một Worker Node
Container lifecycle (CRI) Network plumbing (CNI / kube-proxy) Storage attach (CSI)

4.1 kubelet — Tác nhân chính trên Node

kubelet là tiến trình quan trọng nhất trên Node, hoạt động theo vòng lặp Sync Loop: liên tục so sánh danh sách Pod được gán cho Node này (lấy từ API Server) với trạng thái container thực tế (lấy từ Container Runtime qua CRI), rồi thực hiện hành động cần thiết (tạo, xoá, restart container) để hai bên khớp nhau.

  • PLEG (Pod Lifecycle Event Generator): lắng nghe thay đổi container runtime để cập nhật nhanh.
  • Probes: chạy liveness/readiness/startup probe định kỳ, quyết định restart container hay loại khỏi Service.
  • cAdvisor (built-in): thu thập CPU/RAM/disk/network metric của từng container, phục vụ HPA và kubectl top.
  • Static Pods: đọc manifest từ thư mục local (không qua API Server) — cách Control Plane tự "tự host" chính nó.

4.2 kube-proxy — Triển khai Service ở tầng mạng

kube-proxy không phải là một proxy theo nghĩa truyền thống (traffic không đi qua tiến trình này). Nó chỉ theo dõi Service/EndpointSlice và sinh ra rule ở kernel để traffic được điều hướng trực tiếp:

ModeCơ chếĐặc điểm
iptables (default cũ)Chuỗi NAT rule, chọn ngẫu nhiên 1 trong N PodO(n) rule lookup — chậm khi nhiều Service
IPVSHash table trong kernel, hỗ trợ nhiều thuật toán LBO(1) lookup — phù hợp cluster lớn
nftables (mới)Thay thế iptables, hiệu năng tốt hơnĐang dần thành default

4.3 Container Runtime & CRI

CRI (Container Runtime Interface) là gRPC API chuẩn giữa kubelet và runtime, gồm 2 service chính: RuntimeService (quản lý Pod sandbox & container) và ImageService (pull/list image). Runtime phổ biến (containerd, CRI-O) lại gọi OCI runtime (runc, gVisor, kata) để tạo namespace, cgroup và thực thi container thật.

05Pod — Đơn vị triển khai nhỏ nhất

Pod không phải là "1 container" — nó là nhóm container chia sẻ network namespace và (tuỳ chọn) volume.

Sơ đồ 5 — Cấu trúc bên trong một Pod
Tất cả container trong Pod dùng chung 1 địa chỉ IP (do "pause container" giữ network namespace), nên 2 container không thể cùng lắng nghe 1 port. Init Container chạy lần lượt và phải hoàn tất trước khi App Container bắt đầu — dùng để chuẩn bị môi trường (chờ dependency, clone config...).

06Luồng đầy đủ: từ kubectl apply đến container chạy

Đây là sơ đồ "xương sống" — tổng hợp tất cả thành phần trên thành một luồng request/response thực tế.

Sơ đồ 6 — Sequence: Tạo một Pod từ đầu đến cuối

Điểm mấu chốt: không có thành phần nào "gọi trực tiếp" thành phần khác. Mọi giao tiếp đều theo mô hình watch-based eventual consistency qua API Server — đây là lý do Kubernetes chịu lỗi tốt (một controller crash và restart sẽ tự "đọc lại" toàn bộ trạng thái rồi tiếp tục).

07Mô hình Network — 4 nguyên tắc nền tảng

#1 Mọi Pod có 1 IP riêng (flat)

Mỗi Pod nhận 1 IP từ dải IP của cluster — không NAT giữa Pod, dù chúng nằm trên Node khác nhau.

#2 Pod thấy IP của mình giống bên ngoài thấy

Không có "translation" địa chỉ ở tầng Pod — đơn giản hoá rất nhiều cho service discovery.

#3 Node giao tiếp được mọi Pod

kubelet/healthcheck trên Node bất kỳ có thể reach Pod ở Node khác (qua route do CNI thiết lập).

#4 Container trong cùng Pod chia sẻ network

Như đã thấy ở Sơ đồ 5 — giao tiếp qua localhost, không qua NAT.

Sơ đồ 7 — Traffic giữa Pod trên 2 Node khác nhau (CNI overlay)
Tuỳ CNI plugin: overlay network (Flannel VXLAN) đóng gói packet gốc trong UDP để vượt qua hạ tầng vật lý, hoặc routed network (Calico BGP, Cilium native routing) quảng bá route Pod CIDR trực tiếp — nhanh hơn nhưng yêu cầu hạ tầng hỗ trợ.

08Service & DNS — Service Discovery

Pod là "phù du" (ephemeral) — IP thay đổi mỗi khi Pod restart. Service giải quyết bài toán này bằng cách cung cấp 1 VIP (Virtual IP) ổn định + DNS name, rồi load-balance tới Pod hiện tại.

Sơ đồ 8 — Service ClusterIP hoạt động như thế nào
EndpointSlice Controller liên tục đồng bộ danh sách IP của các Pod khớp selector và đang Ready (qua readiness probe). kube-proxy nhận danh sách này và cập nhật rule kernel — do đó việc "loại Pod lỗi khỏi Load Balancer" xảy ra tự động, không cần can thiệp.
Loại ServicePhạm vi truy cậpCách hoạt động
ClusterIPNội bộ clusterVIP chỉ route được trong cluster
NodePortBên ngoài, qua IP:Node-PortMở port cố định (30000-32767) trên MỌI Node
LoadBalancerInternetcloud-controller-manager tạo LB của Cloud Provider, trỏ vào NodePort
ExternalName—CNAME DNS, không tạo VIP, chỉ ánh xạ tên

09Storage — PV / PVC / StorageClass / CSI

Sơ đồ 9 — Dynamic Provisioning với CSI
  • PV (PersistentVolume): đại diện 1 ổ đĩa thật (vòng đời độc lập với Pod).
  • PVC (PersistentVolumeClaim): "đơn yêu cầu" của Pod — khai báo dung lượng/loại cần.
  • StorageClass: template định nghĩa cách tạo PV (provisioner nào, loại disk nào).
  • CSI: chuẩn plugin để Kubernetes nói chuyện với mọi hệ thống storage mà không cần code cứng vào core.

10Reconciliation Loop — Triết lý lõi của Kubernetes

Mọi controller trong Kubernetes (built-in hoặc Custom Controller/Operator) đều theo cùng 1 khuôn mẫu vô hạn:

Sơ đồ 10 — Control Loop tổng quát
Observe → Diff → Act → Wait — vòng lặp này chạy vô hạn và độc lập với event ban đầu. Nếu một admin sửa tay 1 Pod (vd xoá), controller sẽ không hỏi tại sao — nó chỉ thấy actual != desired và tạo lại Pod mới. Đây chính là nguồn gốc của tính self-healing trong Kubernetes.
