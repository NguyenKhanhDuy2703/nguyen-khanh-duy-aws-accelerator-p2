Layer 1 — Buổi sáng

GitOps & ArgoCD

ArgoCD là một reconciliation loop — desired state đến từ Git. Mọi thay đổi không qua Git đều bị ghi đè.

⟳ ArgoCD Reconcile Loop
🗃️
Git = Source of Truth
Git repo chứa toàn bộ Kubernetes manifests. ArgoCD poll repo 3 phút/lần. Desired state KHÔNG nằm trong cluster — nằm trong Git. Đây là điểm khác biệt cốt lõi với kubectl apply truyền thống.
🔄
Reconcile vs Deploy
ArgoCD không "deploy" — nó reconcile. Nó liên tục so sánh desired (Git) vs actual (cluster) và apply delta. Giống HPA điều chỉnh replicas, nhưng scope toàn bộ cluster.
📌
3 Trạng thái ArgoCD
Synced ✓ — YAML cluster = Git
Healthy ✓ — Pods running OK
Synced ✓ + Degraded ✗ — YAML đúng nhưng image lỗi → Pod crash. Đây là trạng thái nguy hiểm nhất vì ArgoCD không phát hiện được.
⚠️
Vấn đề: Ai quản lý Application object? Nếu bạn kubectl apply -f argocd-apps/ thủ công thì đã phá vỡ GitOps — chính Application object không được quản lý bởi Git.
App-of-Apps Pattern
# k8s-api/argocd/apps/root.yaml apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: root namespace: argocd spec: source: repoURL: https://github.com/you/k8s-api path: argocd/apps # ← folder chứa các app con destination: server: https://kubernetes.default.svc syncPolicy: automated: selfHeal: true prune: true
Quy trình đúng sau khi có root
1. Tạo file YAML app mới
2. Đặt vào argocd/apps/
3. git add . && git push
4. Root App phát hiện → tạo App con tự động
5. App con sync → deploy workload

Không bao giờ kubectl apply trực tiếp!
💥
Lỗi thiếu sync-wave: Deployment apply TRƯỚC ConfigMap → Pod start nhưng không đọc được config → crash. Kubernetes apply tất cả resources cùng lúc nếu không có ordering.
Sync Wave Order — NS→CM→Deploy→Svc
# Namespace — wave 0 (first) metadata: annotations: argocd.argoproj.io/sync-wave: "0" --- # ConfigMap — wave 1 metadata: annotations: argocd.argoproj.io/sync-wave: "1" --- # Deployment — wave 2 metadata: annotations: argocd.argoproj.io/sync-wave: "2" --- # Service — wave 3 (last) metadata: annotations: argocd.argoproj.io/sync-wave: "3"
🪤
Bẫy selfHeal kinh điển: Bạn chạy kubectl rollout undo để rollback khẩn cấp. Nhưng trong ~3 phút, ArgoCD poll Git → phát hiện drift → ghi đè lại bản lỗi từ Git. Rollback của bạn biến mất.
❌
Sai — kubectl rollout undo
  1. kubectl rollout undo deployment/api
  2. Pod rollback thành công ✓
  3. 3 phút sau ArgoCD poll...
  4. Drift detected: cluster ≠ Git
  5. selfHeal: apply lại Git (bản lỗi!)
  6. 💥 Undo bị ghi đè, lỗi quay lại
✅
Đúng — git revert → push
  1. git revert HEAD (hoặc sửa image tag)
  2. git push origin main
  3. ArgoCD poll phát hiện change
  4. ArgoCD apply bản cũ từ Git
  5. ✓ Rollback thành công, Git là truth
Layer 2 — Buổi chiều phần 1

Observability Stack

SLI đo từ góc độ người dùng. Error budget biến cảm tính thành số. Burn rate cảnh báo trước khi budget cạn.

Luồng thu thập metrics — Flask → Prometheus → Grafana
📋
SLA — Service Level Agreement
Hợp đồng với khách hàng. Vi phạm → đền tiền, mất uy tín. VD: "99% uptime/tháng". Đây là cam kết pháp lý, không phải technical target.
🎯
SLO — Service Level Objective
Mục tiêu nội bộ, chặt hơn SLA để có buffer. VD: "99.5% success rate / 30 ngày". Nếu SLA = 99%, SLO = 99.5% → có 0.5% buffer trước khi breach SLA.
📏
SLI — Service Level Indicator
Số đo thực tế từ user perspective: success_req / total_req. Không phải CPU usage! CPU là resource metric, không đo user experience.
🔗
Quan hệ: SLI đo thực tế → so với SLO (target nội bộ) → nếu vi phạm SLO liên tục → có nguy cơ vi phạm SLA (hợp đồng) → đền tiền.

Error Budget — Tính toán từ SLO
💡
Error budget là công cụ ra quyết định: Còn budget → team có thể deploy, thử nghiệm. Hết budget → dừng mọi thay đổi rủi ro, focus vào reliability. Biến tranh luận cảm tính "deploy hay không?" thành quyết định bằng số.
Burn Rate — Tốc độ tiêu thụ Error Budget
Error Rate Burn Rate Budget cạn trong Mức độ
0.5% 1× 30 ngày (đúng kế hoạch) NORMAL
5% 10× 3 ngày WARN
7.2% 14.4× ~2 ngày CRITICAL
⚡
Fast window: 1h × 5min
Bắt sự cố cháy nhanh. Nếu burn rate cao trong 1h, check lại 5 phút trước để xác nhận — tránh false positive do spike ngắn.
🐌
Slow window: 6h × 30min
Bắt degradation âm thầm. Lỗi 5% trong 6h liên tục là vấn đề nghiêm trọng hơn lỗi 20% trong 10 phút rồi hết.

SLI query — Success rate

# Tỷ lệ request thành công 5 phút gần nhất sum(rate(flask_http_request_total{ status=~"2.." }[5m])) / sum(rate(flask_http_request_total[5m]))

Burn rate query (dùng trong AnalysisTemplate)

# Burn rate cho Argo Rollouts sum(rate(http_requests_total{ status=~"2.." }[2m])) / sum(rate(http_requests_total[2m])) # Threshold: >= 0.95 → PASS # < 0.95 (3 lần) → ABORT

ServiceMonitor CRD

apiVersion: monitoring.coreos.com/v1 kind: ServiceMonitor metadata: name: flask-app spec: selector: matchLabels: app: flask-api endpoints: - port: http path: /metrics interval: 15s
✅
ServiceMonitor là cầu nối CRD: thay vì sửa prometheus.yml thủ công, khai báo một ServiceMonitor object → kube-prometheus-stack tự động cấu hình scrape target.
Layer 3 — Buổi chiều phần 2

Canary & Argo Rollouts

Argo Rollouts hijack quá trình rollout của Deployment, kiểm soát traffic từng bước và tự động abort dựa trên metrics.

Canary Deployment Flow — 4 bước tự động
Traffic weight progression
25%
setWeight: 25
25% traffic → v2. 75% vẫn ở v1. Argo Rollouts tạo canary ReplicaSet riêng.
⏸️
pause: {}
Chờ promote (thủ công) hoặc AnalysisTemplate tự động quyết định.
50%
setWeight: 50
Tăng lên 50/50. Tiếp tục theo dõi metrics. AnalysisRun chạy song song.
✅
setWeight: 100
Promoted! 100% traffic → v2. Canary ReplicaSet scale down, stable ReplicaSet update.
🧠
AnalysisTemplate là chiếc cầu nối hai tầng: chạy ở tầng control (Argo Rollouts) nhưng query data từ tầng dữ liệu (Prometheus) để đưa ra quyết định tự động về traffic.
apiVersion: argoproj.io/v1alpha1 kind: AnalysisTemplate metadata: name: success-rate spec: metrics: - name: success-rate interval: 30s # query Prometheus mỗi 30s successCondition: result[0] >= 0.95 # ≥ 95% → pass failureLimit: 3 # 3× fail → ABORT tự động provider: prometheus: address: http://prometheus:9090 query: | sum(rate(http_requests_total{status=~"2.."}[2m])) / sum(rate(http_requests_total[2m]))
≥ 95% — Tiếp tục
AnalysisRun trả về Successful. Rollout tiếp tục bước tiếp theo.
< 95% × 1-2 lần
Tăng failure counter. Argo Rollouts pause, chờ thêm dữ liệu.
< 95% × 3 lần → ABORT
AnalysisRun = Failed. Rollout tự động abort, traffic về v1.

✅

25% → metrics ≥ 95% ✓

100% → Promoted ✓

25% → metric tụt < 95%

Infrastructure Deep Dive

Hạ tầng bên dưới

Những gì xảy ra trong Kubernetes khi ArgoCD, Argo Rollouts và Prometheus hoạt động cùng nhau.

K8s Internal Architecture — Full Stack
🗄️
etcd — nguồn sự thật của K8s
Tất cả objects (Pods, Services, Deployments, CRDs) đều được lưu trong etcd dưới dạng key-value. kube-apiserver là gateway duy nhất đọc/ghi etcd. Khi ArgoCD "apply" manifest, thực chất là ghi vào etcd qua apiserver.
📡
Watch mechanism
Controllers không poll etcd định kỳ — chúng watch (long-poll). Khi etcd thay đổi, apiserver push event đến tất cả watchers. Đây là cơ chế cốt lõi của reconcile loop — event-driven, không phải polling.
# Khi ArgoCD apply một Deployment: # 1. ArgoCD → kubectl apply → apiserver # 2. apiserver validate → write etcd # 3. etcd → event → ReplicaSet controller # 4. RS controller → tạo Pods # 5. Scheduler → assign Pods → Nodes # 6. kubelet trên Node → pull image → start container # Khi Argo Rollouts hijack Deployment: # Rollouts controller watch Rollout CRD # Tạo 2 ReplicaSets: stable + canary # Điều chỉnh replicas từng bước # Sửa Service selector để route traffic # etcd path: /registry/argoproj.io/rollouts/default/flask-api /registry/apps/replicasets/default/flask-api-stable /registry/apps/replicasets/default/flask-api-canary

Canary Traffic Routing
Argo Rollouts điều chỉnh traffic bằng cách thay đổi số lượng replicas, không phải IP rules. setWeight: 25 với 4 replicas = 1 canary pod + 3 stable pods. Traffic được phân phối theo pod count qua kube-proxy/iptables.
🔀
Service → Pod routing
Rollouts tạo 2 Services: stable-svc và canary-svc. Thay đổi selector của main Service để route traffic. Với Istio/Nginx ingress → weight-based routing chính xác hơn.
# Canary weight = 25%, 4 replicas: # stable ReplicaSet: 3 pods # canary ReplicaSet: 1 pod # iptables round-robin → ~25% # Prometheus scrape flow: # prometheus-operator watch ServiceMonitor # → update prometheus config # → scrape /metrics từ Pod IPs # → lưu vào TSDB (time-series DB) # → AnalysisTemplate query PromQL # Service discovery flow: ServiceMonitor (selector: app=flask-api) → matches Service (flask-api) → Endpoints (Pod IPs: 10.0.x.x) → Prometheus scrape http://10.0.x.x/metrics

Hạ tầng chuyên sâu — 5 sơ đồ

Giải phẫu hệ thống từng tầng

5 sơ đồ độc lập, mỗi cái "mổ xẻ" một lớp: etcd watch loop, request lifecycle, canary traffic split, Prometheus scrape pipeline, và toàn bộ data flow end-to-end.

① etcd Watch Loop — Cơ chế cốt lõi của mọi reconcile
② Request Lifecycle — Từ git push đến Pod chạy trong cluster
③ Canary Traffic Split — Argo Rollouts quản lý 2 ReplicaSet trong cluster
④ Prometheus Scrape Pipeline — Từ Pod /metrics đến AnalysisTemplate decision
⑤ End-to-End Data Flow — Toàn bộ pipeline W9 trong 1 sơ đồ
etcd: nguồn sự thật duy nhất
Mọi object K8s đều tồn tại trong etcd. Controller không giao tiếp trực tiếp với nhau — chúng chỉ đọc/ghi etcd qua apiserver. Đây là lý do hệ thống scale được: thêm controller mới không cần sửa code controller cũ.
Watch > Poll: event-driven
Controller không poll định kỳ như cron job. Chúng dùng HTTP/2 long-lived watch stream — etcd push event ngay khi có thay đổi. Latency từ "git push" đến Pod chạy thường < 30s (không tính pull image).
CRD: K8s extensibility
ArgoCD, Rollouts, Prometheus Operator đều dùng CRD để thêm resource types mới. K8s lưu chúng vào etcd y hệt built-in objects. Controller của từng tool watch CRD của nó — không phụ thuộc K8s core.
Common Pitfalls

Lỗi hay gặp & cách xử lý đúng

Những bẫy kinh điển mà engineer mới gặp khi làm việc với GitOps pipeline.

Lỗi #1
kubectl apply tay → selfHeal ghi đè
Bất kỳ thay đổi không qua Git đều bị ArgoCD revert trong vòng 3 phút. Fix: Luôn thêm file vào Git rồi push. Dùng argocd app diff để kiểm tra trước khi sync.
Lỗi #2
Canary không có AnalysisTemplate → người phải canh
Thiếu AnalysisTemplate → pause: {} chờ mãi mãi hoặc phải promote thủ công. Fix: Luôn gắn AnalysisTemplate với Rollout spec. Nếu chưa có Prometheus, dùng web metric hoặc job provider.
Lỗi #3
Thiếu sync-wave → Deployment trước ConfigMap → crash
Kubernetes apply tất cả resources song song nếu không khai báo wave. Fix: Annotation argocd.argoproj.io/sync-wave theo thứ tự: 0=NS, 1=CM/Secret, 2=Deploy, 3=Svc, 4=Ingress.
Lỗi #4
SLI = CPU usage → không đo user experience
CPU thấp không đồng nghĩa users hài lòng. Fix: SLI phải đo từ user perspective: request success rate, latency p99. Dùng flask_http_request_total thay vì container_cpu_usage_seconds.
Lỗi #5
Rollout CRD không trong Git → ArgoCD không biết
Nếu Rollout.yaml không được commit vào k8s-api repo, ArgoCD sẽ không deploy nó. Fix: Cấu trúc repo rõ ràng: k8s-api/apps/flask-api/rollout.yaml phải tồn tại trong Git.
